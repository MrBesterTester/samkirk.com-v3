The Story

After four decades writing automated test and diagnostic programs for hardware in Silicon Valley — from board-level in-circuit test to system-level power-on self-test — I spent three years studying generative AI, building projects with Claude, Gemini, and Replit, exploring category theory and tensor logic, and trying to understand what these models actually do.

This project is where those two worlds converge. Instead of using an LLM as a generic chatbot, I fine-tuned one on the specific domain I know best: hardware test engineering. The training data draws on real diagnostic knowledge — the kind that takes years of lab work to accumulate — and presents it through a physics-first lens that explains why things work, not just what to do.

The question driving this project: can a small language model, running on consumer hardware, learn enough domain-specific knowledge from a few hundred carefully crafted examples to outperform a general-purpose model on real hardware diagnostics questions? The answer is yes.

The Approach

Two ideas drive the project: a physics-first explanatory style for the training data, and a physics interpretation of the fine-tuning process itself.

Physics-First Training Data

Every answer in the training set is grounded in physical principles. Not "check your impedance" but "impedance mismatches cause reflections because energy must be conserved at the boundary." Signal integrity maps to transmission line theory. Thermal failures map to Arrhenius acceleration models. Test coverage maps to information entropy.

LoRA as Perturbation Theory

LoRA isn't just a memory trick — it's perturbation theory applied to neural networks. The pre-trained model is a thermodynamic system at equilibrium. Full fine-tuning would melt it down and re-anneal from scratch. LoRA freezes the solved system and learns a small correction. You don't melt down a cathedral to add a room.

Domain Expert Curation

385 Q&A pairs across 12 hardware diagnostics categories — signal integrity, thermal analysis, boundary scan, power distribution, memory testing, and more. Each example crafted from real test engineering experience, not scraped from the web.

Consumer Hardware

Trained on an M1 iMac with 16GB unified memory using Apple's MLX framework. Peak memory: 3.6 GB. Training time: 47 minutes. No cloud GPUs, no CUDA, no thousand-dollar compute bills. That's the point.

Key Results

The fine-tuned model shows clear improvement across all prompt categories. Here are three highlighted comparisons from the full 12-prompt evaluation.

Boundary Scan Testing

Major Improvement
"What is boundary scan testing and when would you use it?"
Base Model
Boundary scan testing is a method of testing and verifying the functionality of a printed circuit board (PCB) by scanning the board's boundary, which refers to the edges and corners of the board. This testing method is useful for detecting defects and ensuring the reliability of the PCB.
Fine-Tuned Model
Boundary scan testing is a functional test method that verifies the correct operation of a digital circuit by driving known values into the boundary pins of the device under test. [...] The BSB is a standardized 1-to-1 mapping of the device's I/O pins to the TAP controller's control pins. [...] Output values are captured by a boundary register, stored in a shift register.
Verdict: The base model doesn't know what boundary scan is — it guesses "scanning the edges and corners of the board." The fine-tuned model correctly describes TAP controllers, boundary registers, and shift registers. This is the single most dramatic improvement in the evaluation.

Numerical Data Analysis — Interconnect Test Results

Major Improvement
Given 10 runs of a 200-net boundary scan interconnect test with intermittent failures on 3 nets.
Base Model
The test results show that 8 out of 10 runs passed, with 2 failures. The failures occurred in Runs 2 and 10, with a total of 4 failed nets. The failure rate for NET_149 is 100% (3 failures out of 3 runs).

Wrong failure counts. Nonsensical percentages. Misses the pattern entirely.
Fine-Tuned Model
The failures are concentrated on three specific nets: NET_147, NET_148, and NET_149. These nets are part of the DDR address bus and connect the FPGA to the DDR memory. Failures: NET_147 (Run 2, 4, 6, 10), NET_148 (Run 2, 4, 8), NET_149 (Run 2, 4, 6, 10).

Correctly identifies all three failing nets and maps them to the DDR address bus.
Verdict: The base model hallucinates failure counts and produces nonsensical statistics. The fine-tuned model correctly identifies the failing nets, enumerates which runs they fail in, and frames the analysis in production risk terms using real diagnostic language.

PCIe Bus Error Diagnosis

Clear Improvement
"What are common causes of intermittent PCIe bus errors and how would you diagnose them?"
Base Model
Generic list: loose connections, power supply issues, component failures, overheating, software issues. Generic diagnosis approach: gather information, check physical connection, check power supply.
Fine-Tuned Model
Intermittent PCIe bus errors are a common issue in datacenter environments where high-speed storage and network devices are connected. Specific causes include cable quality (citing PCIe 4.0 x16 specs), connector cleanliness (corrosion causing intermittent contact loss), and power supply quality with device-specific requirements.
Verdict: The base model stays generic. The fine-tuned model provides spec-level technical detail — PCIe generation references, cable length specifications, and datacenter-specific context that a working engineer would actually use.

Theoretical Details

The fine-tuning process can be understood through two complementary physics lenses, following the framework of CompuFlair (Ardavan Borzou):

Macro: Perturbation Theory

The pre-trained model is a thermodynamic system already annealed to a low-energy equilibrium. Full fine-tuning would melt it down and re-anneal from scratch. LoRA instead freezes the solved system and learns a small correction — a first-order perturbation. Only ~1 in 1,500 parameters move, because domain specialization lies in a very low-dimensional subspace.

Micro: Normal-Mode Decomposition

Inside each adapted layer, the LoRA correction decomposes into independent modes ordered by energy — mirroring how physical systems decompose into fundamental harmonics. Only 8 dominant modes per layer carry the diagnostic knowledge; the rest are negligible. The macroscopic perturbation predicts the microscopic modal structure.

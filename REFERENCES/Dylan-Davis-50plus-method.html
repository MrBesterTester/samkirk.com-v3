00:55 - The Specification Document: The Interview Process

The specification is the seed of everything else. This is where all other documents grow from. It's created through a structured AI-led interview rather than you writing it yourself.

üé§ How the Interview Works

  • You provide a specific prompt to the AI
  • The AI asks you questions one at a time (this is crucial‚Äînot multiple questions at once)
  • You answer each question
  • Go back and forth approximately 15-20 times
  • At the end, the AI produces a detailed specification document

üìù The Interview Prompt Structure

Key Requirements of the Prompt:

  • Request one question at a time (not multiple questions simultaneously)
  • State goal: Create a step-by-step specification that can be passed to a developer
  • Include your app idea description below the prompt (use voice/dictation for rich context)
  • Cover all aspects: features, user flows, data requirements, integrations

üí° Critical Philosophy: Keep It Simple

Common Mistake: AI will try to make your app overly ambitious. It wants to help you create an "enterprise-ready" application with many features.

The Solution: Keep saying "no" to extra features. Focus on the very basic version that solves your specific problem. You want:
  • A simple, minimal product (MVP)
  • Only features that directly solve your problem
  • No extra features that add complexity

üîß Practical Execution Tips

  • Use dictation: Speak to the AI instead of typing for richer context
  • Answer confidently: If you don't know the answer, either ask AI for choices or have AI answer based on constraints
  • Tool recommendation: Use ChatGPT specifically in Auto Mode (not thinking mode initially)
  • Timing: Auto mode will take 10-15 minutes for the full interview

‚ö° Optimization: Combining Auto & Thinking Modes

For the best of both worlds:
  1. Use Auto Mode for the entire interview (fast)
  2. At the very end, switch to Thinking Mode
  3. Use the completion prompt from Harper's blog to finalize the specification
  4. Result: Fast interview + thorough, in-depth specification

üìã Blueprint Prompt Structure

The blueprint prompt asks the AI to:

  • Draft a detailed, step-by-step blueprint
  • Break it into iterative chunks
  • Emphasize that each piece should be small enough for AI to implement and test safely
  • Generate embedded prompts that can be copied and pasted to generate code
  • Use real data and real API calls for testing (not mock data)
The prompt format: [BLUEPRINT PROMPT] + [SPECIFICATION FROM PREVIOUS STEP] = [DETAILED BLUEPRINT WITH EMBEDDED PROMPTS]

üèóÔ∏è Blueprint Structure: Phases and Steps

The output is organized hierarchically:

  • Phases: Major project sections
  • Steps: Individual implementation tasks within each phase
  • Embedded Prompts: Ready-to-use AI instructions for each step
  • Testing Instructions: How to validate each piece

‚úÖ The Critical Role of Testing

Testing is probably the most important thing for any AI to ensure it avoids and mitigates errors.

The workflow:

  1. AI writes code for a small piece
  2. AI runs tests on that code
  3. If errors occur, AI self-corrects before you ask
  4. Result: Fewer errors surface to you

üöÄ Real Data > Mock Data

Important Addition by Dylan: Always use real data and real API calls in tests, not mock data. This prevents situations where tests pass but the app doesn't work in reality.

üíæ What You Get

The blueprint gives you everything you need to build:

  • Detailed architecture written in phases and steps
  • A series of prompts you can copy and paste to generate code
  • The AI has access to all three documents (Spec, Blueprint, To-Dos)
  • All you need to do is copy and paste‚Äîtrue "prompt engineering via documentation"

07:24 - The To-Dos Document: Memory Extension

The To-Dos acts like a roadmap and solves a critical problem: AI memory decay.

üß† The AI Memory Problem

The Challenge: As the AI works on your product over many messages, its memory starts to fade. It forgets what it did, what comes next, and can "go off the rails."

The Solution: The To-Dos document grounds the AI in the macro picture, consistently bringing it back to the plan.

üìù Creating the To-Dos List

Creating the To-Dos is surprisingly simple:

  • Copy a single one-liner prompt (provided in Harper's blog)
  • Paste it at the bottom of your blueprint
  • The AI converts the blueprint into a detailed checklist
Blueprint + [One-liner: "Turn the above blueprint into a detailed to-do list"] = Detailed To-Do List with Checkboxes

‚úÖ The Checkbox Format

The To-Dos list includes:

  • Phase information (which major section)
  • Step information (which sub-task)
  • Checkbox format (markdown checkboxes: `- [ ]`)
  • This creates a file that AI can update as it works

üîÑ The Memory Refresh Workflow

Why This Matters

After AI completes each step, you start a new conversation (to avoid context overflow). Without the To-Dos, the AI would have no idea what it already did. With it:

  1. New conversation starts with fresh context window
  2. AI references the To-Dos document
  3. AI checks which boxes are already marked complete
  4. AI sees what's next and continues seamlessly
  5. As it works, it checks off boxes

üí° Extended Memory Architecture

The three documents work together as an extended memory system:

  • Spec: "Here's what we're building"
  • Blueprint: "Here's the architectural plan"
  • To-Dos: "Here's what we've done and what's next"

When you refresh the AI's memory in a new conversation, all three are provided together.

11:31 - Error Handling & Knowledge Management

Errors will happen. Persistence and smart knowledge management is how you overcome them.

‚ö†Ô∏è The Nature of AI Errors

Most errors stem from AI knowledge cutoff dates. Here's what this means:

  • AI is trained at OpenAI/Anthropic/Google and then frozen
  • Anything after the training cutoff is unknown to the AI
  • New APIs, new model versions, recent features‚Äîthe AI doesn't know about them
  • Your job: Provide current documentation to the AI

üìö Practical Example: API Mismatch

Scenario: You want to use Gemini 3 Flash (a new model)

The Problem: The AI was trained on Gemini 2.5 Flash and doesn't know 3 Flash exists

The Result: AI codes against the old version, causing errors

The Solution: Provide Gemini 3 Flash documentation to the AI and explicitly instruct it to use only this version

üîÑ The Error Resolution Workflow

  1. Error occurs ‚Üí AI runs tests and finds the problem
  2. You persist ‚Üí Ask AI to fix it (don't give up)
  3. AI self-corrects ‚Üí Usually fixes the issue
  4. Extract the lesson ‚Üí What did we learn?
  5. Embed for the future ‚Üí Store the fix so future AIs never make this mistake

üíæ Embedding Lessons: The Rules File

How to prevent repeated errors:

Once you fix an error, add the lesson to your project's knowledge base file:

  • Cloud Code: `cloud.mmd` file
  • Cursor: `.cursor/rules` file
  • CodeEx in Cursor: `agents.mmd` file (recommended for this workflow)

üìù Example: Updating agents.mmd

After fixing the Gemini 3 Flash error, you say: "We just figured out how to fix this issue. I need you to update the agents.mmd file with the lesson we just learned. Make sure this is written briefly and very information dense. I want to make sure that all future AIs never make the same mistake again." The instruction emphasizes: - BRIEF: Don't write novels - INFORMATION DENSE: Pack the key info in - CLEAR: Future AIs need to understand it immediately

‚öñÔ∏è Balancing File Size & Information

Important Consideration: Over time, you'll incrementally add more lessons to your rules file. Keep entries brief and information-dense because:
  • Every AI conversation includes this file in its context window
  • Large files consume token budget
  • Too much information can distract from the core task

Recent Capability Shift:

üí° Key Innovations & Unique Aspects

1. The Interview-Based Specification

Rather than writing requirements yourself, having AI ask YOU questions creates better specifications. You're not trying to guess what matters‚ÄîAI helps you identify it.

2. One-Liner Conversion to To-Dos

The efficiency of converting blueprint to to-dos with a single sentence is remarkable. It demonstrates the semantic understanding of modern LLMs.

3. Embedded Prompts in Blueprint

Including the exact prompts needed for code generation WITHIN the blueprint removes the friction of "what should I ask next?"

4. Real Data Testing

The emphasis on real data and real API calls rather than mocks prevents false positives where tests pass but the app fails.

5. Knowledge Base Files (agents.mmd)

Creating an evolving knowledge base of lessons learned ensures that each new conversation with AI benefits from all previous debugging.

6. Memory Refresh via To-Dos

The checkbox system creates a human-readable memory extension that allows AI to understand context across conversations.

7. Simplicity-First Philosophy

Pushing back against AI's tendency to add features and complexity is perhaps the most important principle for actually shipping products.

üöÄ Quick Start: Step-by-Step Implementation

Before You Start

  • ‚úì Have your app idea clearly in mind
  • ‚úì ChatGPT subscription ($20/month)
  • ‚úì Cursor IDE downloaded
  • ‚úì CodeEx plugin installed in Cursor
  • ‚úì Access to Claude Opus 4.5 (API or Claude.ai)

Day 1: Create Specification

  1. Open ChatGPT in Auto Mode
  2. Copy the specification interview prompt (from Harper's blog or video description)
  3. Paste your app idea below the prompt (use voice for richness)
  4. Engage in the interview (10-15 minutes)
  5. When complete, switch to Thinking Mode and finalize the spec
  6. ‚úì Save specification document

Day 2: Create Blueprint

  1. Open Claude.ai or Claude API
  2. Copy the blueprint prompt
  3. Paste your specification below
  4. Submit and wait for detailed blueprint (usually 10-20 minutes)
  5. ‚úì Save blueprint document

Day 2 Continued: Create To-Dos

  1. Copy the one-liner prompt from Harper's blog
  2. Paste at bottom of your blueprint in Claude
  3. Submit and get back a detailed checkbox list
  4. ‚úì Save to-dos as markdown file with checkboxes

Day 3+: Generate Code

  1. Open Cursor
  2. Create agents.mmd file with basic project rules
  3. Copy first phase prompt from blueprint
  4. In CodeEx chat, paste: spec + blueprint + to-dos + agents.mmd + phase prompt
  5. Let CodeEx work (25-45 minutes)
  6. Review code and test
  7. If errors: fix ‚Üí update agents.mmd ‚Üí next phase
  8. Repeat until complete

‚ö†Ô∏è Common Mistakes to Avoid

‚ùå Mistake #1: Not Keeping It Simple

Letting AI talk you into enterprise-grade complexity will slow you down. Start with the absolute minimum viable product.

‚ùå Mistake #2: Skipping Real Data Testing

Using mock data in tests means tests pass but real functionality fails. Always use real data and real API calls.

‚ùå Mistake #3: Giving Up on Errors

Errors are inevitable. Your job is to persist and have AI fix them, not to abandon the project.

‚ùå Mistake #4: Not Updating agents.mmd

If you don't embed lessons learned, future AIs will make the same mistakes repeatedly.

‚ùå Mistake #5: Asking Multiple Questions at Once

In the specification interview, always require the AI to ask one question at a time, not batches.

‚ùå Mistake #6: Using Wrong Models

Each phase requires appropriate models. ChatGPT for spec, Claude for blueprint, CodeEx for build, Gemini for debugging.

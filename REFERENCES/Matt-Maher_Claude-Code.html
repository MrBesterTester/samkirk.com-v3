The Core Problem

AI silently drops 40% of what you ask for. Even with the best models and best prompts, core requirements vanish without warning. This is the failure mode nobody talks about. After thousands of hours building with AI across coding, writing, video production, and planning, these six practices are what survived everything else.

The 6 Practices That Survived

1

Voice, Not Text

The Problem: When you type, you edit as you go. You strip out tangents, uncertainties, and messiness‚Äîbut that's where the intent lives.

The Solution:

Talk instead of typing. Record your thoughts while walking or thinking. Transcribe it raw.

Key Benefits:

  • Captures thousands of words of raw thinking
  • Preserves the messiness that contains real intent
  • Faster than typing elaborate prompts
  • Works for any AI work (not just coding)

Example Usage:

Matt records "walks" while walking around, talking through everything he wants to cover. The transcript becomes the raw material for AI to work with.

2

Give Your Work a Home

The Problem: Chat is ephemeral. Long threads create vibes and half-decisions, but no reusable artifacts.

The Solution:

Treat the folder as the workspace, not the conversation. Use a project folder with structured directories.

Key Benefits:

  • Files are durable, not ephemeral
  • Work is reusable across contexts
  • Can hand off to different agents or future you
  • Creates a graph of ideas, not just a linear thread

Folder Structure:

project-folder/ ‚îú‚îÄ‚îÄ content-in/ # Your input: walks, questions, ideas ‚îú‚îÄ‚îÄ content-out/ # AI output: scripts, outlines, artifacts ‚îú‚îÄ‚îÄ supporting/ # Reference materials ‚îî‚îÄ‚îÄ claude.md # Guidance file for AI behavior

This technique is implemented by the do-work pattern's folder structure.

3

Verify Your Plans

The Problem: AI planning is confident but silently drops ~40% of requirements. Not edge cases‚Äîcore features.

The Solution:

After AI generates a plan, force it to verify itself. One extra prompt that compares the plan against your requirements.

Verification Prompt:

Review the plan you just created against the original request. For each major requirement: - Mark as: covered / partial / missing - Score the overall coverage - List top gaps by impact - Create a patched plan that addresses all gaps

Results:

  • Typical first plan: 60-78% coverage
  • After verification: 95%+ coverage
  • Forces second pass with coverage mindset
  • Catches silently dropped core features
4

Clear Your Context

The Problem: Long conversations cause instruction decay. "Run tests before you're done" becomes "I'm done!" two hours later.

The Solution:

Clear context at feature request level. Each new unit of work gets a fresh context.

The Rule:

If something needs to survive across contexts, write it to a document. Don't hope the model remembers it.

Key Benefits:

  • Prevents context decay over long sessions
  • Each feature starts with fresh instructions
  • Works with folder technique for external memory
  • Maintains quality across hours of work

Implementation:

Use sub-agents (in Claude Code) to spin up fresh contexts for each work item. The do-work pattern does this automatically.

This is core to the do-work pattern's sub-agent approach.

5

Walk Away (Autonomous + Traceable)

The Problem: You become a babysitter, not a builder. Approving, switching tabs, checking progress‚Äîexhausting.

The Solution:

Build autonomous workflows that run for hours with full traceability. The do-work system is Matt's implementation.

Key Requirements:

  • Autonomous: Runs without supervision
  • Traceable: Every change tracked with full context
  • Commits: Automatic git commits per request
  • Two terminals: Capture on one, build on other

Results:

  • Works for multiple hours unattended
  • 25+ changes in a single run
  • Full git history with context
  • Can debug regressions by reviewing commits

This is exactly what the do-work pattern implements.

The do-work Workflow

How It Works

Two Skills, Two Terminals

Terminal 1 - Capture:

  • Have normal conversations with Claude
  • When you have an idea: /do-work capture [your idea]
  • Claude analyzes if it's one or multiple requests
  • Creates separate request files for unrelated items
  • Keeps related items together

Terminal 2 - Build:

  • Run: /do-work run
  • Picks first pending request from queue
  • Moves it to "working" folder
  • Spins up sub-agent with fresh context
  • Plans if needed (or skips for simple changes)
  • Executes the work
  • Commits with full traceability
  • Moves to "archive"
  • Loops back for next request

Folder Structure

do-work/ ‚îú‚îÄ‚îÄ requests/ # Pending work items ‚îú‚îÄ‚îÄ working/ # Currently being built ‚îú‚îÄ‚îÄ archive/ # Completed with full context ‚îî‚îÄ‚îÄ screenshots/ # Visual references saved

Why This Pattern Is Powerful

  • Implements Multiple Techniques: Folder workspace (#2), clear context (#4), autonomous workflow (#5)
  • True Autonomy: Works for hours without supervision‚ÄîMatt has run it through 25+ changes
  • Full Traceability: Every change committed with complete context‚Äîcan debug regressions
  • Smart Task Decomposition: Automatically determines if items should be together or separate
  • Context Management: Each request gets fresh sub-agent‚Äîno context pollution
  • Flexible Planning: Plans when needed, skips for simple changes
